# 设计书

## 设计摘要

- 推荐方案: 分阶段迁移，先建立统一 Store/Repository 抽象并保留 SQLite 默认行为，再把 API key、账号、配置、异步任务/结果等真实源从 JSON/文件迁入数据库；外部数据库优先实现 PostgreSQL，MySQL 放到第二阶段。
- 推荐原因: 当前项目持久化由 SQLite、JSON 文件、本地任务文件、本地/HF 图片文件、日志文件共同组成，不能只替换 `sqlite3.connect` 或加 `DATABASE_URL`；PostgreSQL 与现有 `ON CONFLICT`/JSON payload 更接近，适合先落地。
- 入口模块: `api/app.py`、`api/ai.py`、`api/async_jobs.py`、`services/job_service.py`、`services/metadata_db.py`、`services/config.py`、`services/account_service.py`、`services/api_key_service.py`、`services/data_service.py`。
- 关键调用链: 请求进入 FastAPI 后由 `api/ai.py` 或 `api/async_jobs.py` 调用 `JobService`；`JobService` 读写 `jobs/*.json`、`job_results/*.json` 并同步 `MetadataDatabase`；设置/账号/API key 分别由 `ConfigStore`、`AccountService`、`APIKeyService` 读写 JSON 文件；图片由 `data_service.save_image_bytes()` 写入本地或 HF datasets 并把 URL/path 交给任务结果和画廊元数据。
- 数据流: 配置/API key/账号/CPA/Sub2API 配置目前先进入 JSON 文件；异步任务先进入 job/result JSON；任务、画廊、瀑布墙、会话、请求日志再进入 SQLite 元数据表；图片 bytes 不进入 DB，而是进入本地文件或 HF datasets。
- 错误处理: 当前 SQLite 有本地 corrupt quarantine；外部 DB 需要改为连接失败、迁移失败、事务冲突、唯一键冲突、超时、连接池耗尽等显式错误；不能吞错造成伪成功。
- 配置变更: 后续实现才建议新增 `CHATGPT2API_DB_BACKEND`、`CHATGPT2API_DATABASE_URL`、连接池参数、迁移 CLI 参数；本轮不改配置。
- 用户是否批准: 已批准本轮只读检查和报告交付；实现阶段需用户另行下达开始迁移指令。
- 批准范围: 本轮范围仅限只读源码调查、迁移方案分析、DevOne 任务包文档沉淀；不包含业务源码修改、数据库结构变更、生产数据迁移或远程数据库连接。

## 方案对比

### 方案 A

分阶段 Store 抽象 + PostgreSQL 优先 + MySQL 后续。

- 做法: 先抽象 `MetadataStore`、`JobStore`、`SettingsStore`、`AccountStore`、`APIKeyStore`，默认 backend 仍是 SQLite/JSON；随后逐步迁移 JSON/任务文件真实源；最后实现 PG backend 和导入工具。
- 优点: 风险最低，可保持现有单实例行为；能逐项验证；避免 MySQL/PG 方言同时扩散到业务层。
- 缺点: 工期比“直接替换 sqlite3”长。
- 结论: 推荐。

### 方案 B

直接把 `services/metadata_db.py` 改造成多方言 SQL。

- 做法: 在现有类中加入 DB URL 判断，分支处理 SQLite/PG/MySQL SQL。
- 优点: 表面改动集中。
- 缺点: 不能解决 API key、账号、配置、任务 JSON、图片/日志文件这些真实源；会快速堆积方言分支；多实例仍不可靠。
- 结论: 不推荐。

## 分阶段计划

### 阶段 1

锁定持久化接口边界，不改变现有行为。

- 产出: Store 接口、SQLite/JSON 默认实现、配置开关草案、契约测试骨架。
- 验证: 现有测试全部通过；默认启动仍使用 `/app/data`。

### 阶段 2

迁移真实源到数据库，并实现 PostgreSQL backend。

- 产出: API key、账号、配置、异步任务、任务结果的 DB 真实源；PG schema migration；SQLite/JSON 导入 CLI。
- 验证: SQLite 回归、PG 单实例、PG 多实例、迁移对账测试通过。
## 结论

当前项目不能很好地直接迁移到外部 MySQL 或 PostgreSQL；不能只加一个 `DATABASE_URL` 就完成迁移。

原因不是“数据表太复杂”，而是持久化边界没有统一：核心元数据在 SQLite，配置/账号/API key/导入配置仍以 JSON 文件为真实源，异步任务运行依赖本地 `jobs/*.json`、`job_results/*.json`、`task_logs/*.log`，图片实体仍依赖本地文件或 HF datasets。外部数据库只能解决其中一部分问题。

可行路线是分阶段改造：先抽象持久化接口，再把 JSON 源和异步任务源迁入数据库，最后实现 PostgreSQL/MySQL backend 和迁移工具。

## 当前持久化架构

```text
HTTP/API 层
  ├─ api/app.py: 启动时 record_settings / record_accounts；中间件写 request_logs
  ├─ api/ai.py: OpenAI 兼容图片接口可创建 inline job，并关联任务/画廊/瀑布墙
  └─ api/async_jobs.py: 任务、画廊、瀑布墙、会话、SSE 事件接口

服务层
  ├─ services/metadata_db.py: SQLite 元数据表
  ├─ services/job_service.py: jobs/result JSON + metadata_db 双写/回填
  ├─ services/config.py: config.json 真实配置源 + settings_snapshots 镜像
  ├─ services/account_service.py: accounts.json 真实账号源 + metadata_db 镜像
  ├─ services/api_key_service.py: api_keys.json 真实 API key 源
  ├─ services/cpa_service.py: cpa_config.json
  ├─ services/sub2api_service.py: sub2api_config.json
  └─ services/data_service.py: local/HF datasets 图片实体 + 清理统计
```

## 外部数据库适配难点

### 1. SQLite 方言深度耦合

证据：`services/metadata_db.py`

- 直接 `import sqlite3`，构造 `sqlite3.connect(self.path)`。
- 使用 `sqlite3.Row`，调用方依赖 `row["field"]` 行访问。
- 初始化使用 `PRAGMA quick_check`、`PRAGMA journal_mode=WAL`。
- 表探测使用 `sqlite_master`。
- 增量列探测使用 `PRAGMA table_info(...)`。
- 主键使用 `INTEGER PRIMARY KEY AUTOINCREMENT`。
- Upsert 使用 SQLite/PostgreSQL 风格 `ON CONFLICT(...) DO UPDATE SET ... excluded...`，MySQL 不兼容。
- 参数占位符全部为 `?`，PostgreSQL/MySQL Python driver 通常不是这个占位符。
- 本地损坏隔离逻辑会移动 `metadata.sqlite3`、`metadata.sqlite3-wal`、`metadata.sqlite3-shm`，外部 DB 不适用。

### 2. 真实源仍分散在 JSON 文件

外部 DB 即使替换 `metadata_db.py`，仍不能覆盖：

- `services/config.py`: `config.json` 仍是配置真实源；`settings_snapshots` 只是快照。
- `services/account_service.py`: `accounts.json` 仍是账号真实源；`metadata_db.record_accounts()` 是镜像。
- `services/api_key_service.py`: `api_keys.json` 保存 client key、hash、权限、配额、计数。
- `services/cpa_service.py`: `cpa_config.json` 保存 CPA pool 和导入 job 状态。
- `services/sub2api_service.py`: `sub2api_config.json` 保存 Sub2API server 和导入 job 状态。

### 3. 异步任务仍以文件参与运行

证据：`services/job_service.py`

- `_job_file(job_id)` -> `jobs/<job_id>.json`
- `_result_file(job_id)` -> `job_results/<job_id>.json`
- `_task_log_file(...)` -> `task_logs/<timestamp>_<job_id>.log`
- `_read_json()` / `_write_json()` 读写任务和结果。
- `submit_job()` 先写 job JSON，再写 metadata，再提交线程池。
- `_store_partial_image_result()` 对批量/流式图片会更新 result JSON，再同步 metadata。
- `get_job_result()` 仍会结合 metadata 和 result 文件读取。

因此如果只迁 `async_jobs` 表，仍会出现 DB 有记录但结果文件/日志文件缺失，或者多实例中某实例看不到另一实例本地文件的问题。

### 4. 图片实体不是关系数据库问题

证据：`services/data_service.py`

- 本地模式写入 `DATA_DIR/images/<date>/<file>` 及缩略图、瀑布墙图。
- HF datasets 模式调用 `_upload_hf_dataset_file(...)` 写入远端数据集。
- 数据库只存 `url`、`relative_path`、`thumbnail_relative_path`、`wall_relative_path` 等元数据。
- 清理统计遍历本地目录，HF datasets 下不等价。

外部 DB 适合保存图片元数据，不适合直接保存图片 bytes；应保留 local/HF datasets，或未来引入 S3/R2/MinIO 一类对象存储。

### 5. 当前没有外部 DB 基础设施

证据：`pyproject.toml`

当前依赖没有 `SQLAlchemy`、`Alembic`、`psycopg`、`asyncpg`、`pymysql`、`mysqlclient` 等数据库抽象/驱动/迁移工具。项目也没有 `DATABASE_URL`、连接池、schema migration CLI 或 repository/provider 抽象。

## MySQL 与 PostgreSQL 适配评估

```text
项目点                         PostgreSQL                  MySQL
SQLite ON CONFLICT             较接近，仍需改占位符/DDL     需改成 ON DUPLICATE KEY UPDATE
JSON payload                   建议 JSONB                   可用 JSON/TEXT，但索引能力不同
布尔字段 0/1                   可映射 boolean/int           可映射 tinyint(1)
AUTOINCREMENT                  bigserial/identity           auto_increment
LIMIT/OFFSET                   兼容                         兼容
LIKE 查询                      兼容                         兼容，需注意 collation
schema introspection           information_schema           information_schema
并发与多实例                   更推荐                       可用，但 upsert/事务语义需细测
迁移优先级                     推荐第一优先                 第二优先
```

建议先做 PostgreSQL。原因：当前 SQL upsert 风格更接近 PostgreSQL，JSONB 能更好承载现有 `payload_json`，并发/多实例部署更稳。MySQL 可以作为第二 backend，但不应与 PG 同波次实现。

## 推荐迁移路线

### Phase 0：只读评估与契约锁定

- 输出当前报告。
- 明确数据库要承担“元数据 + 配置 + 任务状态”，图片实体仍走文件/HF/对象存储。
- 明确兼容目标：SQLite 默认保留；PostgreSQL 优先；MySQL 后续。

### Phase 1：持久化抽象，不改变行为

- 新增 `MetadataStore` / `SettingsStore` / `AccountStore` / `APIKeyStore` / `JobStore` 接口。
- 当前 SQLite/JSON 文件实现作为默认 backend。
- 业务层不再直接依赖 `sqlite3.Connection`、`Path` JSON 文件。
- 配置新增 `CHATGPT2API_DB_BACKEND`、`CHATGPT2API_DATABASE_URL`，但默认仍是 SQLite/JSON。

### Phase 2：迁移 JSON 真实源

优先级：

1. `api_keys.json` -> DB 表：认证、配额、计数必须支持事务/并发。
2. `accounts.json` -> DB 表：账号池状态、成功失败计数、刷新时间。
3. `config.json` -> DB 表或保留 bootstrap 文件 + DB 覆盖：避免“设置保存后重启丢失”。
4. `cpa_config.json` / `sub2api_config.json` -> DB 表：导入配置和导入 job 状态。

### Phase 3：任务运行源迁移

- `jobs/*.json` 和 `job_results/*.json` 改为 DB 表真实源。
- `task_logs/*.log` 可先保留文件，但 DB 需要保存日志索引；如果要多实例查看日志，再追加 `task_log_entries` 表。
- 流式/批量图片的 partial result 必须在 DB 中追加或更新，保证“一张生成就一张可见”。

### Phase 4：实现 PostgreSQL backend

- 选择一种策略：
  - 轻量方案：手写 SQL + driver adapter + migration SQL。
  - 更长期方案：SQLAlchemy Core/ORM + Alembic。
- 建议长期选 SQLAlchemy Core + Alembic，减少 MySQL/PG 双 backend 方言分叉。
- 完成 SQLite schema 到 PostgreSQL schema 的显式迁移脚本。

### Phase 5：MySQL backend

- 在 PG 稳定后再做 MySQL。
- 重点处理 upsert、JSON、大小写/排序规则、datetime、锁和事务差异。

### Phase 6：数据迁移 CLI

- `export-sqlite`：导出 `metadata.sqlite3` + JSON 文件 + 文件引用清单。
- `import-db`：导入 PG/MySQL。
- `verify-migration`：计数、抽样、图片 URL/path、任务状态、API key 权限、账号数量对账。

## 功能域迁移可行性矩阵

```text
功能域                     当前真实源                         外部 DB 迁移难度    结论
系统配置                   config.json + settings_snapshots    中                可迁，但要定义 bootstrap/DB 优先级
API key/配额               api_keys.json                       中高              应优先迁，需事务保证配额计数
账号池                     accounts.json + accounts 镜像       中高              可迁，需避免全量 DELETE/INSERT 竞争
异步任务列表               jobs JSON + async_jobs              高                必须改 JobStore，不能只迁表
任务结果                   job_results JSON + result_json       高                必须 DB 化，否则多实例不可靠
任务日志                   task_logs/*.log + task_logs 表       中高              可先索引，后续再结构化日志表
画廊                       gallery_images                      中                DB 元数据可迁；图片实体仍外部存储
瀑布墙                     gallery_images                      中                同画廊
绘画会话记录               image_conversations                 中                可迁；当前 schema 仍带 api_key_id 但查询已全局化
请求日志                   request_logs                        低中              可迁，注意写入量和清理策略
CPA/Sub2API 配置           cpa_config.json/sub2api_config.json  中                可迁，需迁导入 job 状态
图片文件                   local files / HF datasets           非 DB 问题         不建议入关系库，保留对象/数据集存储
数据清理                   本地目录遍历                         高                DB 后需重写统计/清理策略
```

## 推荐优先级

1. P0：建立 Store 抽象和配置开关，不改行为。
2. P1：API key、账号、配置迁入统一 Store。
3. P1：任务/结果真实源迁入 DB，保持画廊/瀑布墙实时可见。
4. P2：PostgreSQL backend + SQLite 导入工具。
5. P3：MySQL backend。
6. P3：结构化日志表和对象存储 backend。

## 不建议的改法

- 不建议直接把 `services/metadata_db.py` 中 `sqlite3.connect` 替换成 PG/MySQL driver。
- 不建议在原类里大量 `if backend == ...` 拼方言 SQL。
- 不建议把图片 bytes 直接塞进 MySQL/PG。
- 不建议继续让 DB 与 JSON 双写但没有单一真实源，否则重启、多实例、回滚时容易互相覆盖。
