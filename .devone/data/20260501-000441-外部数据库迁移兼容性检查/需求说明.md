## 问题定义

用户要确认“当前项目的所有功能是否能很好地迁移到外部 MySQL 或 PostgreSQL”。这里的“迁移”按工程语义拆成两层：

1. 数据库元数据层迁移：把当前 `metadata.sqlite3` 中的任务、画廊、瀑布墙、会话、请求日志等记录迁到外部数据库。
2. 全部持久化能力迁移：把配置、账号、API key、CPA/Sub2API 配置、异步任务文件、任务结果文件、任务日志、图片实体等也纳入外部数据库或外部存储体系。

## 真实问题判断

这是真问题，不是假设问题。当前项目已经多次围绕“重启后数据丢失、画廊/瀑布墙/任务列表持久化、配置文件路径、HF datasets 图片存储、OpenAI 兼容 API 跟踪”做过改动；现有实现仍是 SQLite + JSON 文件 + 本地/HF 图片文件 + 日志文件的混合持久化模型。部署到容器、HF Space、反代或多实例环境时，外部数据库迁移需求合理且必要。

## 范围

本次检查覆盖：

- `services/metadata_db.py` 的 SQLite schema、查询、写入、初始化、损坏隔离逻辑。
- `services/job_service.py` 的异步任务、任务结果、任务日志、画廊/瀑布墙/会话访问路径。
- `services/config.py` 的配置文件和数据目录策略。
- `services/account_service.py` 的账号 JSON 源。
- `services/api_key_service.py` 的 API key JSON 源和配额计数。
- `services/cpa_service.py` / `services/sub2api_service.py` 的外部导入配置和导入任务状态。
- `services/data_service.py` 的图片实体、本地文件、HF datasets 存储和清理统计。
- `api/app.py` / `api/ai.py` / `api/async_jobs.py` 中的启动记录、OpenAI 兼容接口跟踪、SSE/任务接口调用关系。
- 测试现状中与数据库/持久化相关的覆盖范围。

## 非目标

- 本轮不直接实现 MySQL/PG 后端。
- 本轮不做数据库结构变更。
- 本轮不迁移真实用户数据。
- 本轮不修改生产配置、环境变量或远程仓库。

## 验收标准

- 能明确回答当前项目是否可以“直接切换”到外部 MySQL/PG。
- 能列出阻碍迁移的具体文件、符号和实现点。
- 能按功能域评估迁移难度和风险。
- 能给出分阶段迁移路线、优先级和测试要求。
