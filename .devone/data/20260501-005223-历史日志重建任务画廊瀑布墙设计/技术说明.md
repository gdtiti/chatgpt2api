# 技术说明

## 本轮执行摘要

本轮继续按用户选项 3 推进，完成“历史日志重建任务、历史、画廊、瀑布墙”的只读调查和功能设计，没有修改业务源码。

## 已采集证据

- `web/src/app/gallery/page.tsx`：画廊页前端固定 `limit=20`。
- `web/src/app/wall/page.tsx`：瀑布墙前端固定 `limit=40`。
- `web/src/app/jobs/page.tsx`：任务页默认 `limit=50`，已有分页/筛选/自动刷新。
- `api/async_jobs.py`：后端 gallery 默认 20、wall 默认 40、jobs 默认 50。
- `services/job_service.py`：现有 backfill 只在 metadata 没有任何任务时扫描 `jobs/*.json`，且当前样本 jobs/result 目录为空。
- `utils/log.py`：system/task log 是结构化行日志；base64 图片会被保存到 images 目录并替换为 `[image: URL]`。
- 当前本地 `data/metadata.sqlite3` 表计数：`async_jobs=12`，`gallery_images=0`，`image_conversations=0`，`request_logs=1303`。
- 当前本地 `data/images` 有 57 个文件，说明图片实体仍可能存在。
- 当前本地 `data/system.log` 有可解析结构化事件。

## 设计结论

可以实现“尽力重建”，但必须对恢复结果分可信度：

- metadata/job result JSON 可高可信恢复。
- task log/system.log 可恢复任务状态和部分图片 URL，但不保证 prompt。
- images 目录可恢复画廊/瀑布墙图片索引，但 prompt、账号、模型可能缺失。
