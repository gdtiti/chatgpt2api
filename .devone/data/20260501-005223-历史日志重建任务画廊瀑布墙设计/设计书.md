# 设计书

## 设计摘要

- 推荐方案: 增加“历史重建 / 索引修复”功能，按 `metadata -> job/result JSON -> task logs -> system.log -> images 目录` 的优先级扫描，先 dry-run 生成恢复计划，再 apply 幂等写入 `async_jobs`、`gallery_images`、`task_logs`，并在前端维护页提供扫描、预览、应用和报告。
- 推荐原因: 当前丢失的主要是索引和 metadata，不一定是图片实体；本地 `data/images` 仍有文件，`system.log` 仍有结构化事件。单靠现有 `_backfill_metadata_if_empty()` 不够，因为它只在 DB 没有任何任务时扫描 `jobs/*.json`，且当前 job/result JSON 为空。
- 入口模块: 新增后端 service 建议为 `services/history_recovery_service.py`；新增 API 建议放在 `api/system.py` 或新建 `api/recovery.py`；前端入口建议放在设置页数据维护卡片，必要时增加独立 `/settings/recovery` 区块。
- 关键调用链: 管理员点击“扫描可恢复历史” -> 后端扫描当前 `DATA_DIR` -> 返回 recovery report -> 用户确认 apply -> 后端按幂等规则写入 `metadata_db.record_async_job()` / 新增批量恢复方法 -> 前端刷新任务、画廊、瀑布墙。
- 数据流: 文件/日志只作为只读来源；恢复计划写入内存或临时 report；apply 时只新增/补齐缺失记录，不删除、不覆盖用户状态。
- 错误处理: 单个文件解析失败记入 warnings，不中断全局扫描；数据库写入失败回滚当前批次并返回错误；路径越界、非管理员、非法外部路径直接拒绝。
- 配置变更: 可选新增 `history_recovery_enabled`、`history_recovery_scan_extra_dirs`、`history_recovery_max_files`；第一阶段可以不加配置，只提供管理员手动触发。
- 用户是否批准: 已批准本轮只读设计；实现阶段需用户后续确认“开始开发”。
- 批准范围: 本轮只产出设计文档，不修改业务源码，不执行恢复写库。

## 方案对比

### 方案 A

手动触发的 dry-run/apply 历史重建。

- 优点: 安全、可控、可预览；不会每次启动误导入测试数据或旧数据；适合当前用户反馈。
- 缺点: 用户需要手动点一次或调用 API。
- 结论: 推荐作为第一阶段。

### 方案 B

应用启动时自动扫描并恢复。

- 优点: 用户无感。
- 缺点: 启动变慢；容易误导入测试日志、旧临时目录、重复图片；出现错误影响服务启动。
- 结论: 不推荐默认启用。可在后续作为显式配置 `history_recovery_auto_on_empty=true`。

### 扫描器分层

#### 1. Metadata 扫描器

来源：

- 当前 `DATA_DIR/metadata.sqlite3`
- `metadata.sqlite3.corrupt-*` 中仍可打开的文件
- 用户显式上传/指定的旧 metadata 文件，必须限制路径或复制到安全临时目录后读取

能力：

- 读取 `async_jobs`、`gallery_images`、`image_conversations`、`task_logs`。
- 对当前 DB 缺失的 job/gallery/image_conversation 生成候选。
- 对旧 DB 的 `result_json` 和 `preview_images_json` 生成高可信图片候选。

注意：

- corrupt 文件不一定能打开；失败只记录 warning。
- 不直接 attach 未信任 DB 到当前连接，优先只读单独连接读取。

#### 2. Job JSON 扫描器

来源：

- `DATA_DIR/jobs/*.json`
- `DATA_DIR/job_results/*.json`

能力：

- 比现有 `_backfill_metadata_if_empty()` 更强：即使 metadata 不为空，也补齐缺失 job 或缺失 gallery。
- 不限制 500 条，使用可配置上限和分页扫描。
- 从 result JSON 中抽取 `data[].url` / `b64_json` / markdown 图片。

#### 3. 日志扫描器

来源：

- `DATA_DIR/task_logs/*.log`
- `DATA_DIR/system.log`
- `metadata_db.task_logs.log_path` 指向且仍存在的文件

解析规则：

```text
^<timestamp> [<level>] <json-or-text>$
```

事件合并规则：

- `async_job_submitted` / `inline_job_started` 创建 job 候选。
- `async_job_started` 更新状态为 running。
- `async_job_succeeded` / `inline_job_succeeded` 更新状态为 succeeded，记录 result_count。
- `async_job_failed` / `inline_job_failed` 更新状态为 failed，记录 error。
- `image_task_start` 如果包含 prompt/model/size/quality，用于补齐 payload。
- 日志中的 `[image: URL]`、`image_urls_resolved.urls`、`image_final_urls.urls` 可生成图片候选。

限制：

- 当前 system.log 样本里的 `async_job_submitted` 不含 prompt，因此只能恢复 job_id/model/status/api_key/log_path 等基本信息。
- 如果 task log 文件不存在，不能恢复详情日志，只能保留原 log_path 和 warning。

#### 4. Images 目录扫描器

来源：`DATA_DIR/images/<YYYY-MM-DD>/...`

识别规则：

- 跳过 `*-thumb.*` 和 `*-wall.*` 作为原图候选。
- 原图文件名匹配：`<request_id>-<index>.<ext>`。
- 尝试配对：
  - thumbnail: `<request_id>-<index>-thumb.<ext>`
  - wall: `<request_id>-<index>-wall.<ext>`
- 如果 `<request_id>` 与已有 job_id 匹配，归入该 job。
- 如果没有匹配 job，创建 synthetic job：
  - `job_id = recovered-image-<hash(relative_path)>` 或使用 `request_id`（需避免和真实 job 冲突）。
  - `type = images.recovered`
  - `status = succeeded`
  - `model = recovered`
  - `prompt_preview = None`
  - `payload.recovery_source = images_dir`

注意：

- 不把 10 字节测试图片当成真实图片的判断应谨慎。建议只记录 `warnings: suspicious_small_file`，不默认跳过，避免误删真实小文件。
- 图片 URL 通过现有 `build_image_url()` 生成，保证反代/base_url 规则一致。

## 写入策略

### 幂等原则

- `async_jobs.job_id` 已存在：默认不覆盖，只补齐空字段和缺失 `result_json`/`preview_images_json`。
- `gallery_images(job_id, image_index)` 已存在：保留 `is_recommended/is_pinned/is_blocked`，只补齐 URL/path 空字段。
- 不删除任何现有记录。
- 不清空任何表。

### 需要补充的 metadata_db 能力

现有 `record_async_job()` 会先 `DELETE FROM gallery_images WHERE job_id=?` 再插入，会影响已有状态。恢复功能建议新增专用方法：

```python
metadata_db.recover_async_job(candidate, *, merge_only=True)
metadata_db.recover_gallery_images(job_id, images, *, preserve_state=True)
metadata_db.get_recovery_existing_state(job_ids)
```

或者扩展 `record_async_job()`，加入 `preserve_existing_gallery=True`，但为避免影响正常写入，推荐新增恢复专用方法。

## API 设计

管理员接口：

```text
POST /api/system/recovery/scan
GET  /api/system/recovery/reports/{report_id}
POST /api/system/recovery/apply
```

请求示例：

```json
{
  "sources": ["metadata", "jobs", "logs", "images"],
  "include_corrupt_metadata": true,
  "include_system_log": true,
  "include_task_logs": true,
  "include_images": true,
  "dry_run": true
}
```

返回示例：

```json
{
  "report_id": "20260501T005223Z_xxx",
  "dry_run": true,
  "stats": {
    "jobs_found": 28,
    "jobs_new": 16,
    "jobs_merge": 12,
    "images_found": 19,
    "gallery_new": 19,
    "warnings": 7
  },
  "candidates": [...],
  "warnings": [...]
}
```

`apply` 必须携带 `report_id` 或重新传入 `plan_hash`，防止扫描后文件变化造成误应用。

## 前端设计

入口：设置页增加“历史重建 / 索引修复”卡片。

页面元素：

- 数据源选择：metadata、job/result JSON、task logs、system.log、images 目录。
- 按钮：扫描、应用恢复、下载报告。
- 统计：可恢复任务数、可恢复图片数、将新增画廊项、将新增瀑布墙项、警告数。
- 候选列表：job_id、状态、类型、模型、时间、来源、可信度、警告。
- 图片预览：对 images 目录候选显示缩略图。
- 安全提示：不会删除现有数据；缺失 prompt 不能凭空恢复。

同时改进：

- 画廊页增加 page size 选择：20/50/100。
- 瀑布墙增加 page size 选择：40/80/100。
- 当 total 大于当前页大小时明确显示“当前仅显示本页 N 条，共 M 条”。

## 分阶段计划

### 阶段 1

后端只读扫描 + 报告。

- 新增 `HistoryRecoveryService.scan()`。
- 支持 metadata/job JSON/log/images 扫描。
- 新增管理员 scan API。
- 前端设置页展示 dry-run 报告。

### 阶段 2

幂等 apply + metadata 写入。

- 新增恢复专用 DB merge 方法。
- 支持恢复 async_jobs、gallery_images、task_logs。
- 应用后刷新任务、画廊、瀑布墙页面。

### 阶段 3

体验增强和分页改进。

- 画廊/瀑布墙 page size 选择。
- 恢复报告下载。
- 可选支持扫描旧 metadata 文件上传。

## 风险与缓解

```text
风险                             缓解
误导入测试/临时数据               dry-run + 来源/可信度 + 用户确认 apply
重复导入                         job_id/image_index 幂等约束
覆盖用户置顶/屏蔽状态             preserve_state，不覆盖已有状态
缺失 prompt                      显示“未能从日志恢复”，不伪造
旧 log_path 不存在                保留路径并加 warning，不报成功日志内容
大目录扫描慢                     设置 max_files、分页扫描、后台任务化
路径越界                         只允许 DATA_DIR 和显式白名单路径
```
