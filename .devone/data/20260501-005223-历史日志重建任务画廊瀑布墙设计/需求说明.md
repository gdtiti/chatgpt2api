# 需求说明

## 文档元信息

- 任务名称: 历史日志重建任务画廊瀑布墙设计
- 项目根目录: `J:\_Dev\_toapi\_codex\chatgpt2api-gdtiti`
- 工作流档位: `devone-fast`
- 执行模式: `required-only`
- 详细设计模式: `classic`
- 本轮性质: 只读调查 + 功能设计，不改业务源码。

## 当前问题

用户反馈：目录里有很多 metadata / meta 相关数据，但每次启动后任务、历史、画廊、瀑布墙好像都不见；当前最新页面看起来也只是显示 20 个。希望判断是否能通过历史日志重建，并设计该功能。

## 真实问题判断

这是真问题，现有代码和本地数据都能支撑该判断：

- 当前 `data/metadata.sqlite3` 存在，但 `gallery_images=0`、`image_conversations=0`，`async_jobs=12`，说明元数据不完整。
- 当前 `data/jobs`、`data/job_results` 文件数为 0，因此现有 `_backfill_metadata_if_empty()` 无法从 job JSON 恢复。
- 当前 `data/images` 仍有 57 个图片/缩略图/瀑布墙文件，说明“图片实体还在，但画廊/瀑布墙索引没了”。
- `data/system.log` 有结构化事件，例如 `async_job_submitted`、`async_job_started`、`async_job_succeeded`、`inline_job_started`、`inline_job_succeeded`、`image_generate_slot_start`、`image_slot_return_success`。
- 画廊页面前端硬编码 `limit=20`，所以“最新只显示 20 个”对画廊页是真实 UI 限制；瀑布墙硬编码 `limit=40`；任务页默认 `limit=50`。

## 目标 1

设计一个“历史重建 / 索引修复”功能，从现存数据源重建缺失的任务、画廊、瀑布墙索引。

## 目标 2

设计功能必须支持 dry-run 预览、幂等 apply、不会删除现有数据、可说明每条记录来源和可信度。

## 范围

- 后端扫描并重建：当前 metadata DB、旧/隔离 metadata DB、job JSON、result JSON、task log、system.log、images 目录。
- 前端增加维护入口：扫描、预览、应用、查看报告。
- 页面分页限制改进：画廊/瀑布墙提供分页大小选择，不再让用户误以为只有 20/40 条。

## 非目标

- 不从外部 ChatGPT/OpenAI 服务重新拉取历史。
- 不凭空生成缺失提示词、账号、结果文本。
- 不删除或覆盖现有任务、画廊、瀑布墙记录。
- 不把无法验证的图片强行标记为原始生成结果。

## 业务约束

- 历史记录应尽量全局可见，不再因为 API key / 账号被删除而看不到。
- 恢复结果必须区分 `recovered` 来源，避免和实时任务伪装成完全一样。
- 已存在的用户状态，如推荐、置顶、屏蔽，不能被恢复流程覆盖。

## 技术约束

- 现有主数据库是 SQLite，恢复功能先按当前架构实现。
- 文件扫描必须限制在 `DATA_DIR` 和用户显式配置的路径内，禁止任意路径遍历。
- 日志解析必须容忍坏行、截断 JSON、重复事件、路径不存在。
- 图片扫描必须识别 `-thumb`、`-wall` 派生图，避免把缩略图当作独立原图重复导入。

## 外部依赖

- 无新增外部服务依赖。
- 可选支持用户手动上传或指定旧 `metadata.sqlite3`/日志文件目录，但默认只扫描当前 `DATA_DIR`。

## 本次边界结论

可以设计并实现“尽力重建”：

- 任务列表：可从 metadata、job JSON、system/task logs 较可靠恢复基本状态。
- 画廊/瀑布墙：可从 result JSON 和 images 目录较可靠恢复图片索引。
- 提示词/完整结果：只有 job JSON、result JSON、task log 中存在时才能恢复；system.log 中多数事件不含 prompt，因此不能保证全量恢复。
